# Stub 项目协作约定

## 新会话启动流程

开始任何分析、修改、构建或其他项目操作前，必须依次完成：

1. 阅读 `docs/context.md`，了解当前状态、未验收修改和已知问题。
2. 阅读 `docs/architecture.md`，了解系统结构、数据流和模块边界。
3. 阅读 `docs/decisions.md`，了解已经确定的产品与技术决策。
4. 运行 `git status --short` 和 `git diff`，确认当前工作树状态。

不得覆盖、重置、丢弃或顺手提交用户尚未验收的修改。文档内容与代码现状冲突时，以代码和 Git 状态为事实来源，并在本次工作中修正文档。

## 上下文文档分工

- `docs/architecture.md`：保存稳定架构、模块职责和数据流；只在架构变化时更新。
- `docs/context.md`：保存当前工作状态、待验收修改、已知问题和接手步骤；每次验收、提交或阻塞变化后更新。
- `docs/decisions.md`：保存重要产品和技术决策及原因；产生新决策时追加，改变旧决策时标明替代关系，不静默改写历史。

修改功能时应同步判断这三份文档是否需要更新。不要把临时调试日志写进架构或决策文档；仅把对后续会话仍有价值的信息写入仓库。

## 注释规则

1. 注释解释 Why，不解释 What。
2. 复杂函数开头写明整体思路。
3. 公共 API 使用文档注释（`///`）。
4. 协议、坑点和硬件限制必须写注释。
5. 修改代码时同步更新注释；过期注释直接删除。

## 提交与验收流程

每次实现新的功能、界面调整、打印布局调整或协议行为修改后，必须遵守以下顺序：

1. 先修改代码并完成必要的本地构建、静态检查和单元测试。
2. 将修改结果和验证方式告知用户，等待用户在真机、喵喵机或实际使用流程中验收。
3. 只有用户明确表示“验收通过”“可以提交”“可以 push”或同等意思后，才能执行 `git commit` 和 `git push`。
4. 用户未确认前，不得创建或移动版本标签，不得发布 GitHub Release，不得强制推送历史。

## 未验收时的处理

- 用户反馈问题时，继续在当前工作树修复，并重新构建验证。
- 修复完成后再次等待用户验收，不要因为本地构建成功就自行提交。
- 不要为了保持工作树干净而重置、丢弃或覆盖用户尚未验收的修改。
- 如果用户明确要求立即提交或推送，以用户当前指令为准，并在回复中说明提交内容。

## 交付说明

每次交给用户验收时，应简要说明：改了什么、如何验证、用户需要重点检查什么。提交后应报告 commit hash 和远程推送结果。
