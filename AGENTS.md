# Stub 项目协作约定

## 注释规则

1. 注释解释 Why，不解释 What。
2. 复杂函数开头写明整体思路。
3. 公共 API 使用文档注释（`///`）。
4. 协议、坑点和硬件限制必须写注释。
5. 修改代码时同步更新注释；过期注释直接删除。

## 提交与验收流程

每次实现新的功能、界面调整、打印布局调整或协议行为修改后，必须遵守以下顺序：

1. 先修改代码并完成必要的本地构建、静态检查和单元测试。
2. 将修改结果和验证方式告知用户，等待用户在真机、喵喵机或实际使用流程中验收。
3. 只有用户明确表示“验收通过”“可以提交”“可以 push”或同等意思后，才能执行 `git commit` 和 `git push`。
4. 用户未确认前，不得创建或移动版本标签，不得发布 GitHub Release，不得强制推送历史。

## 未验收时的处理

- 用户反馈问题时，继续在当前工作树修复，并重新构建验证。
- 修复完成后再次等待用户验收，不要因为本地构建成功就自行提交。
- 不要为了保持工作树干净而重置、丢弃或覆盖用户尚未验收的修改。
- 如果用户明确要求立即提交或推送，以用户当前指令为准，并在回复中说明提交内容。

## 交付说明

每次交给用户验收时，应简要说明：改了什么、如何验证、用户需要重点检查什么。提交后应报告 commit hash 和远程推送结果。
